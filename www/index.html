single server

Git push to production in <5 seconds.

The easiest way to run multiple apps on a single server. Install once, add apps, and manage everything from your server.

Use it

Every command runs on your server. Init connects Cloudflare and GitHub. Add apps with their GitHub URL.

SSH to a blank server.

ssh root@203.0.113.10

Install Single Server.

curl -fsSL https://singleserver.com/install.sh | sh

Initialize providers.

singleserver init

Add an app and ship it.

singleserver add https://github.com/you/my-app

Check what is running.

singleserver status

What happens next

GitHub is the trigger. Your server fetches the pushed commit, builds it locally, and hands it to Kamal for a zero-downtime deployment.

git push → GitHub sends a webhook to your server
build → Your server builds the image from that commit
live → Kamal rolls it out with zero downtime

What you get

Fast deploys

No Actions queue, no per-repo runner hop. Build on the server, swap containers, healthcheck, and set commit status.

App domains

The server is managed over SSH. Domains are inferred or added when you add apps.

Tiny Go tool

No Node runtime, no database, no per-repo runner. A small Go binary manages deploys from your server.

Containers per app

Every project runs in its own Docker container, but the host still stays simple and inspectable.

Kamal without boilerplate

Single Server generates deploy config from conventions, while Kamal handles the container rollout.

Monitor from the host

Use status, logs, and doctor commands from the same server where deploys run.

The app contract

Keep repositories boring. A Single Server app only needs a Dockerfile, one web port, and a health endpoint. Node, Bun, static sites, SQLite-backed apps: all fine.

Dockerfile → Your repo defines how it builds
port 80 → Default app port, override when needed
/up → Default healthcheck path

The Single Server command contract.

Single Server itself is a small Go binary. It installs a systemd daemon that listens on 127.0.0.1, keeps apps.yml as the deploy allowlist, receives GitHub push webhooks through Cloudflare Tunnel, checks out the pushed commit with a GitHub App token, renders Kamal config from server-side conventions, and runs the deploy on the same machine. Docker containers, DNS routes, env files, storage mounts, backups, and deploy logs are managed as server resources.

Quickstart

Start with any Linux server from Hetzner, DigitalOcean, AWS, Azure, or another provider. Run every command over SSH on that server.

SSH to your Linux server.

ssh root@203.0.113.10

Install the Single Server CLI.

curl -fsSL https://singleserver.com/install.sh | sh

Initialize your server.

singleserver init

Add your apps.

singleserver add https://github.com/you/my-app

Check what is running.

singleserver status

After that, pushing to the configured branch should deploy automatically.

Concepts

  • Your server The server is both the deploy controller and the runtime host. The daemon, Docker, Kamal, cloudflared, Git checkouts, deploy credentials, app config, persistent storage, backups, and logs all live on this one machine. That is why normal operations happen over SSH instead of across repo settings, CI runners, and separate dashboards.
  • The daemon The daemon listens locally on 127.0.0.1. Cloudflare Tunnel exposes the small public surface it needs: setup pages and GitHub webhooks. When a webhook arrives, the daemon verifies the signature, checks the repository and branch against apps.yml, fetches the exact pushed SHA, runs the deploy, writes logs, and reports the result.
  • GitHub App The GitHub App is how the server receives push events and reads code. It sends signed webhooks to your server, issues short-lived installation tokens for repository fetches, and lets the daemon write pending, success, and failure commit statuses back to GitHub. GitHub triggers the deploy, but the deploy runs on your server.
  • apps.yml This is the central app registry and deployment allowlist. Installing the GitHub App on a repository only grants access; it does not make that repository deployable. A pushed repository must also match an app entry in /etc/singleserver/apps.yml, and the pushed branch must match that app's configured branch.
  • App containers Every app runs in its own Docker container. Single Server prepares the checkout and generated deploy config, then Kamal builds the image, starts the replacement container, runs the healthcheck, and moves traffic only after the new container is ready. The result is container isolation without writing per-app deploy plumbing.
  • Cloudflare Tunnel Public HTTP traffic enters through Cloudflare Tunnel. Cloudflare DNS points app hostnames at the tunnel, and the tunnel config maps each hostname either to the Single Server daemon for setup/webhooks or to the app proxy managed by Kamal. Public TLS terminates at Cloudflare, so generated Kamal config uses plain local HTTP behind the tunnel.
  • Central state Domains, server-side environment files, storage mounts, backups, deploy logs, and provider credentials are server concerns. Repositories stay focused on application code and a Dockerfile. When an app needs an exception, the exception belongs in apps.yml or a server-side command, not in GitHub Actions secrets or a repo-specific runner.

Commands

Commands are grouped by setup, app operations, and server management. Each command owns its arguments, subcommands, and options directly below the command form, so there is no separate options table to cross-reference.

Setup

Install curl -fsSL https://singleserver.com/install.sh | sh

Installs Docker, Kamal, cloudflared, the hosted singleserver Linux binary, the singleserver.service daemon, and base config files. The installer is safe to rerun and can fall back to a source build when explicitly requested.

Initialize your server singleserver init [options]

Runs first-time setup for the server. It creates the base directories and config files, connects or repairs Cloudflare Tunnel when a Cloudflare token is available, restarts the daemon, prints the GitHub App setup URL, and runs diagnostics. GitHub setup is reported as pending until the browser approval is completed.

singleserver init options

  • --zone <domain>Use this Cloudflare zone when the API token can access more than one zone.
  • --skip-cloudflareCreate local Single Server files and GitHub setup state without creating or repairing a Cloudflare Tunnel.
Repair GitHub connection singleserver github connect [options]

Prints the GitHub App setup URL again, creates a setup token if needed, and restarts the daemon so the browser setup page can create or reconnect the GitHub App. The GitHub App needs repository contents read, commit statuses write, and push events.

singleserver github connect options

  • --name <name>Use a custom GitHub App display name, useful for test servers or branded installs.
  • --publicCreate a public/installable GitHub App manifest so the app can be installed under multiple GitHub owners or organizations.
Repair Cloudflare connection singleserver cloudflare connect [options]

Creates a Cloudflare Tunnel, or reuses the configured tunnel when local tunnel credentials are available. It writes cloudflared credentials and config, routes the webhook hostname to the local daemon, stores Cloudflare state, and starts the tunnel service.

singleserver cloudflare connect options

  • --zone <domain>Cloudflare zone to manage. If omitted, Single Server uses the accessible zone when there is only one.
  • --tunnel <name>Cloudflare Tunnel name. Defaults to singleserver.
  • --hook-host <domain>Webhook hostname for the daemon. Defaults to hooks.<zone>.

Apps

Add an app singleserver add https://github.com/owner/repo [options]

Adds a GitHub repository, verifies app access, checks for a Dockerfile, writes apps.yml, validates the generated Kamal config, configures app routing, deploys the current branch tip, runs the healthcheck, and prints the live URL.

singleserver add arguments

  • <github-url>Required. A URL like https://github.com/owner/repo. Single Server stores it as owner/repo.

singleserver add options

  • --branch <name>Deploy only this branch. Defaults to the repository default branch.
  • --app-port <port>Container port used by the generated Kamal proxy. Defaults to 80.
  • --healthcheck-path <path>Path Kamal uses for container healthchecks. Defaults to /up.
  • --healthcheck <url>External URL Single Server checks after deploys and in status/doctor.
  • --name <app>Override the app name. Defaults to the repository name lowercased.
  • --no-deployConfigure the app but wait for the next push or manual deploy.
  • --dry-runValidate the app and print the changes without writing apps.yml.
  • --path <dir>Advanced checkout path override. Defaults to /srv/repos/<app>.
  • --host <domain>Advanced domain override. Can be repeated. Most apps use the inferred app domain or add domains later with singleserver domains add.
Deploy manually singleserver deploy <owner/repo|app> [ref]

Deploys a configured app immediately. If ref is omitted, it deploys the configured branch or the repository default branch. Manual deploys use the same checkout, build, rollout, healthcheck, and GitHub status path as push-triggered deploys.

singleserver deploy arguments

  • <owner/repo|app>Configured repository or app name to deploy.
  • [ref]Optional branch, tag, or commit SHA. If omitted, Single Server uses the app's configured production branch.
List apps singleserver list

Prints configured apps, their repositories, branches, domains, healthcheck URLs, and current health state.

Check status singleserver status

Summarizes daemon health, app deploy status, app domains, runtime state, and healthchecks. This is the quick operational view.

Run diagnostics singleserver doctor [app]

Runs deeper checks for daemon health, config validity, deploy user SSH, local registry, GitHub App access, generated deploy config, checkout state, recent deploy outcome, disk, Docker, Cloudflare Tunnel routes, Cloudflare DNS targets, resolver DNS, and healthchecks.

View logs singleserver logs [app] [options]

Shows deploy logs by default. Filter by app name when provided, use --runtime for app container logs, --daemon for the full Single Server journal, and --follow to stream.

singleserver logs options

  • [app]Optional app name or repository. Without it, Single Server shows recent deploy logs for all apps.
  • --runtimeShow logs from the app container instead of the Single Server daemon journal. Requires an app.
  • --daemonShow the full Single Server service journal instead of only deploy lines.
  • --followStream new log lines instead of printing the recent tail.
Inspect deploy config singleserver render-deploy <owner/repo|app>

Prints the generated Kamal deploy.yml for a configured app. Use this when you want to see exactly what conventions produced before a deploy.

Remove an app singleserver remove <app> [options]

Removes an app from config, stops containers, removes routes, and removes matching DNS records. Repo checkouts and persistent storage are kept by default and deleted only with explicit confirmation.

singleserver remove options

  • --delete-storage --yesDelete the app's persistent storage directory after removing config and stopping containers.
  • --delete-repo --yesDelete the app's repository checkout after removing config and stopping containers.

Management

Manage domains singleserver domains <subcommand> ...

Adds, removes, lists, and verifies app domains. Domain changes update central config, Cloudflare DNS, tunnel routing, and redeploy the app so Kamal picks up the new proxy config.

singleserver domains subcommands

  • add <app> <domain>Add the hostname to the app, create or update the Cloudflare CNAME, add the tunnel route, set a default external healthcheck if one is missing, and redeploy the app.
  • remove <app> <domain>Remove the hostname from the app, remove the matching Cloudflare DNS/tunnel route, clear a matching default healthcheck, and redeploy the app.
  • list [app]Print all configured domains, optionally scoped to one app.
  • verify [app]Check resolver DNS, Cloudflare CNAME targets when available, and Cloudflare Tunnel routes, optionally scoped to one app.

singleserver domains add/remove options

  • --no-deployUse with add or remove to update config and Cloudflare routing without redeploying immediately.
Manage environment singleserver env <subcommand> <app> ...

Sets, lists, and removes app environment variables and secrets on the server. Secrets stay out of GitHub repositories and are injected into the app on the next deploy.

singleserver env subcommands

  • set <app> KEY=valueWrite or replace one app environment value in the server-side env file, then print the deploy command that will apply it.
  • list <app>Print environment values currently stored for the app.
  • unset <app> KEYRemove one app environment value from the server-side env file, then print the deploy command that will apply the removal.
Enable storage singleserver storage enable <app> [options]

Creates persistent storage for SQLite databases, uploads, and other app-owned files, updates app config, and redeploys the app so Kamal mounts it into the container.

singleserver storage enable options

  • --mount <path>Container mount path. Defaults to /storage.
  • --path <path>Host storage path. Defaults to /srv/storage/<app>.
  • --no-deployStage the storage config without applying it to the running app immediately.
Back up an app singleserver backup <app>

Creates a timestamped .tar.gz backup from the app's configured persistent storage. SQLite databases are copied through SQLite's backup API before the archive is written.

Restore an app singleserver restore <app> <backup-id-or-path> --yes [options]

Restores an app backup after explicit confirmation, replaces the storage directory, keeps the previous storage directory as a rollback copy, and restarts the app containers with the restored data.

singleserver restore options

  • --yesRequired. Confirms that the current storage directory should be replaced.
  • --no-restartRestore files without stopping or restarting app containers. Use only when you have already stopped the app yourself.
Upgrade Single Server singleserver upgrade

Downloads the latest release, restarts the service, and runs singleserver doctor to verify the server after the upgrade.

Configuration

Single Server is convention over configuration. The normal path is to add an app with only its GitHub URL. From that one input, Single Server infers the app name, checkout path, deploy branch, app domain, healthcheck path, generated Kamal config, and the default storage location when storage is enabled.

The conventions are meant to be predictable rather than hidden. /etc/singleserver/apps.yml is the central override file when an app needs to differ from the defaults, such as a non-default branch, multiple domains, a different container port, a custom healthcheck, or a storage mount.

Conventions

  • Repository A GitHub URL like https://github.com/owner/repo is normalized to owner/repo in apps.yml. The URL form is the user-facing command shape; the compact form is the stored config shape.
  • App name The app name defaults to the repository name, lowercased. It must be unique on the server because it becomes the stable identifier used for the checkout path, generated Kamal service name, image name, container names, logs, storage paths, and inferred domains.
  • Branch Deploys default to the repository default branch. Set branch only when production deploys should follow a different branch, such as main, master, or production.
  • Domain If Cloudflare is connected, the default app domain is inferred from the app name and connected zone, such as my-app.example.com. Characters that are awkward in hostnames, such as dots or underscores, are replaced with hyphens for the inferred domain. Extra domains are added with singleserver domains add. Single Server writes the Cloudflare CNAME, updates the tunnel route, and rejects hostnames already assigned to another app.
  • Checkout Repositories are checked out under /srv/repos/<app>. Before each deploy, Single Server fetches the exact pushed SHA so production matches the GitHub event that triggered it.
  • Build The app repository owns its Dockerfile. Single Server verifies the Dockerfile exists and uses it, but it does not generate application build instructions.
  • Kamal config Single Server generates Kamal config from conventions and overrides. A repo-level config/deploy.yml should only be needed for unusual apps that opt out of the generated config path.
  • Port Apps listen on port 80 by default. Use --app-port or app_port when the container listens somewhere else, such as a Node or Bun app on 3000.
  • Healthcheck The default container healthcheck path is /up. Kamal uses that path to decide when the replacement container is ready, and Single Server uses the public app domain plus that path as the external healthcheck unless an app sets a different healthcheck URL.
  • Secrets Secrets live on the server in app env files. They are injected into generated Kamal config without requiring GitHub repository secrets, deploy keys, or GitHub Actions workflows.
  • Storage Persistent app data defaults to /srv/storage/<app> and mounts into the container at /storage. This is the expected place for SQLite databases, uploads, and other app-owned data.
  • Backups Backups are written under /srv/backups/<app>. Storage-enabled apps can be backed up and restored from the server without adding app-specific backup code to the repository.
  • Logs Deploy logs and daemon logs live in the systemd journal for singleserver.service. Runtime logs come from the app containers and are surfaced through singleserver logs <app> --runtime.

Apps Config

The shortest app entry is just a repository name.

apps:
  - dvassallo/sillyface-games
  - dvassallo/fullsend

Use an object only when an app needs overrides.

apps:
  - repo: smallbets/userbase-homepage
    branch: master
    hosts:
      - userbase.com
      - www.userbase.com
    app_port: 80
    healthcheck_path: /up
    healthcheck: https://userbase.com/up
  • repoRequired. GitHub repository as owner/repo. This is the only field required for a conventional app.
  • nameDefault: repository name. Lowercase app identifier used for checkout, service, image, container names, logs, storage, and inferred domains. App names must be unique; set this when two repositories share the same repo name.
  • branchDefault: repository default branch. Only pushes to this branch deploy automatically. Manual deploys can still name a specific ref.
  • pathDefault: /srv/repos/<app>. Checkout directory on the server.
  • hostsDefault: DNS-safe app label plus the connected Cloudflare zone. Public hostnames for generated Kamal proxy config, Cloudflare DNS, and Cloudflare Tunnel routing. Hosts must be unique across apps.
  • app_portDefault: 80. Container port that receives web traffic from Kamal.
  • healthcheck_pathDefault: /up. Container healthcheck path used by generated Kamal config.
  • healthcheckDefault: app domain plus healthcheck path. External healthcheck URL used by Single Server after deploys and in status checks.
  • storageDefault: absent until enabled. Persistent storage mapping with optional path and mount fields.

Server Environment

  • SINGLESERVER_CONFIGDefault: /etc/singleserver/apps.yml. Main app config path.
  • SINGLESERVER_STATE_DIRDefault: /etc/singleserver. Directory for GitHub App credentials, Cloudflare state, setup tokens, and Single Server state.
  • SINGLESERVER_PORTDefault: 8787. Local daemon port. The daemon listens on 127.0.0.1 and Cloudflare Tunnel forwards webhook/setup traffic to it.
  • SINGLESERVER_BACKUP_DIRDefault: /srv/backups. Root directory for app storage backups.
  • SINGLESERVER_PUBLIC_URLManaged by init. Public URL used in GitHub webhook setup, normally https://hooks.<zone>.
  • SINGLESERVER_SETUP_TOKENManaged by init. Token used to protect provider setup pages while GitHub and Cloudflare are being connected.
  • SINGLESERVER_GITHUB_APP_NAMEDefault: Single Server. GitHub App display name used by the setup page.
  • SINGLESERVER_GITHUB_APP_PUBLICDefault: private app. Set by singleserver github connect --public when the GitHub App needs to be installable under multiple owners.

Generated Kamal Defaults

  • Service and imageDefault: app name. The generated Kamal service, image, and container identity are derived from the Single Server app name.
  • Server hostDefault: 127.0.0.1. Kamal deploys back to the same machine where Single Server is running.
  • SSHDefault: deploy user with /root/.ssh/id_ed25519. The daemon runs locally, while Kamal keeps using its normal SSH-based deploy model.
  • RegistryDefault: 127.0.0.1:5555. Images are built and pushed to the local registry so deploys do not wait on an external registry round trip.
  • BuilderDefault: local Docker builder for the server architecture. The server builds the image from the checked-out commit.
  • ProxyDefault: ssl: false, forwarded headers on, app port 80. Cloudflare Tunnel handles public TLS; Kamal routes traffic to the app container.
  • HealthcheckDefault: /up, interval 1, timeout 1. Fast local healthchecks keep zero-downtime swaps quick for small apps.
  • TimeoutsDefault: deploy 10s, drain 1s. Defaults are tuned for fast local builds and small app containers.

App Contract

Your apps need a Dockerfile, one web port, and a health endpoint. They do not need GitHub Actions workflows, deploy keys, repo-level deploy secrets, repo-level webhooks, or a Kamal config.

Dockerfile → Your repo defines how it builds
port 80 → Default app port, override with --app-port
/up → Default healthcheck path, override with --healthcheck-path

Dockerfile Examples

Static site:

FROM nginx:alpine
COPY dist /usr/share/nginx/html

Node or Bun app:

FROM oven/bun:1
WORKDIR /app
COPY package.json bun.lock ./
RUN bun install --frozen-lockfile
COPY . .
ENV PORT=3000
EXPOSE 3000
CMD ["bun", "start"]
singleserver add https://github.com/you/my-node-app --app-port 3000

Deploys

Whenever you push to GitHub, your server fetches the commit, builds it locally, and uses Kamal to roll it out with zero downtime.

git push → GitHub sends a webhook to your server
allowlist → apps.yml decides whether this repo can deploy
checkout → Your server fetches the exact pushed SHA
build → Kamal builds locally with generated or repo-provided deploy config
live → Kamal rolls it out and Single Server checks health
status → GitHub gets pending, success, or failure commit status

Files On Your Server

  • /etc/singleserver/apps.ymlApp allowlist and central app config.
  • /etc/singleserver/singleserver.envDaemon environment, including local port, public webhook URL, setup token, and provider setup values.
  • /etc/singleserver/github-app.jsonGitHub App metadata, including app ID and webhook secret.
  • /etc/singleserver/github-app.private-key.pemGitHub App private key used to request short-lived installation tokens.
  • /etc/singleserver/cloudflare.jsonCloudflare zone, tunnel, and routing state created by singleserver cloudflare connect.
  • /srv/repos/<app>Git checkout for each app.
  • /srv/storage/<app>Persistent storage for SQLite databases, uploads, and app-owned files.
  • /srv/backups/<app>Backups created by singleserver backup.
  • journalctl -u singleserver.serviceDaemon and deploy logs.

Build Order

  • Blank server setupinstall.sh and singleserver init should make a new Linux server ready in one guided flow.
  • GitHub and Cloudflare connectioninit should wrap provider setup. github connect and cloudflare connect should exist as repair commands.
  • App onboardingsingleserver add https://github.com/owner/repo should infer sensible defaults, configure routing, deploy immediately, and show the live URL.
  • Domainssingleserver domains should add, remove, list, and verify app domains through Cloudflare Tunnel.
  • Observabilitystatus, doctor, and logs should make deploy state, runtime logs, DNS, proxy, disk, Docker, and healthchecks obvious.
  • Secrets and storageenv, storage, backup, and restore should make SQLite-backed apps practical on one server.
  • Lifecycleremove and upgrade should be first-class commands with clear confirmation and rollback paths.

The Single Server command contract.

Single Server itself is a small Go binary: no Node runtime, no database, and no per-repo runner. It uses Docker, Kamal, and Cloudflare Tunnel on your Linux server to deploy your apps.

Quickstart

Start with any Linux server from Hetzner, DigitalOcean, AWS, Azure, or another provider. Run every command over SSH on that server.

SSH to your Linux server.

ssh root@203.0.113.10

Install the Single Server CLI.

curl -fsSL https://singleserver.com/install.sh | sh

Initialize your server.

singleserver init

Add your apps.

singleserver add https://github.com/you/my-app

Check what is running.

singleserver status

After that, pushing to the configured branch should deploy automatically.

Concepts

  • Your server The control plane. Single Server, Docker, Kamal, cloudflared, checkouts, config, secrets, and deploy logs live here.
  • GitHub App The event source and deploy credential provider. Push events wake your server; installation tokens fetch code and set commit status.
  • apps.yml The allowlist. Only repositories in this file can deploy, even if the GitHub App is installed more broadly.
  • App containers Each project runs in its own Docker container. Kamal builds, starts, healthchecks, and swaps traffic.
  • Cloudflare Tunnel The expected public ingress path. App domains route through Cloudflare to your server without direct TLS management on the host.
  • Central secrets Deploy secrets belong on the server, not in every GitHub repo. Repos should only need app code and a Dockerfile.

Commands

Setup

Command What it does
Install curl -fsSL https://singleserver.com/install.sh | sh Installs Docker, Kamal, cloudflared, the singleserver CLI, the singleserver.service daemon, and base config files. It should be safe to rerun.
Initialize your server singleserver init Connects Cloudflare Tunnel and GitHub, creates required secrets under /etc/singleserver, starts the daemon, and finishes with singleserver doctor.
Repair GitHub connection singleserver github connect Opens or prints a GitHub URL, requests contents read and commit statuses write, and keeps apps.yml as the deployment allowlist.
Repair Cloudflare connection singleserver cloudflare connect Creates or reuses a tunnel, saves credentials, and prepares DNS/routing for app domains.

Apps

Command What it does
Add an app singleserver add <github-url> [options] Adds a GitHub repository, verifies access, checks for a Dockerfile, writes apps.yml, validates the generated Kamal config, and deploys the current branch tip.
Deploy manually singleserver deploy <owner/repo> [ref] Deploys a configured app immediately. If ref is omitted, it deploys the configured branch or the repository default branch.
List apps singleserver list Prints configured apps, their repositories, branches, domains, and health state.
Check status singleserver status Summarizes daemon health, app deploy status, domains, runtime health, and healthchecks.
Run diagnostics singleserver doctor [app] Runs deeper checks for daemon health, config validity, GitHub App access, generated deploy config, checkout state, recent deploy outcome, disk, Docker, proxy, DNS, and healthchecks.
View logs singleserver logs [app] [--follow] [--runtime] Shows deploy logs by default. Use --runtime for app container logs and --follow to stream.
Inspect deploy config singleserver render-deploy <owner/repo> Prints the generated Kamal deploy.yml for a configured app. Useful for debugging conventions.
Remove an app singleserver remove <app> Removes an app from config, stops containers, removes routing, and asks before deleting persistent storage.

Add Options

Option What it does
<github-url> Accepts a URL like https://github.com/owner/repo and normalizes it to owner/repo in config.
--branch <name> Deploys only this branch. Defaults to the repository default branch.
--app-port <port> Container port used by the generated Kamal proxy. Defaults to 80.
--healthcheck-path <path> Path Kamal uses for container healthchecks. Defaults to /up.
--healthcheck <url> External URL Single Server checks after deploys and in status/doctor.
--name <app> Overrides the app name. Defaults to the repository name lowercased.
--no-deploy Configures the app but waits for the next push or manual deploy.
--dry-run Validates the app and prints the changes without writing apps.yml.
--path <dir> Advanced checkout path override. Defaults to /srv/repos/<app>.
--host <domain> Advanced host override for generated Kamal proxy config. Can be repeated. Most domain work should use singleserver domains.

Management

Command What it does
Manage domains singleserver domains add <app> <domain> Adds, removes, and lists app domains. Updates central config, Cloudflare DNS, and tunnel routing.
Manage environment singleserver env set <app> KEY=value Manages app environment variables and secrets on the server.
Enable storage singleserver storage enable <app> --mount /storage Enables persistent storage for SQLite databases, uploads, and other app-owned files.
Back up an app singleserver backup <app> Creates an app backup. SQLite apps should have a first-class, low-drama backup path.
Restore an app singleserver restore <app> <backup-id> Restores an app backup after confirmation.
Upgrade Single Server singleserver upgrade Downloads the latest release, restarts the service, and runs singleserver doctor.

Configuration

The main config file is /etc/singleserver/apps.yml. The shortest app entry is just a repository name.

apps:
  - dvassallo/sillyface-games
  - dvassallo/fullsend

Use an object only when an app needs overrides.

apps:
  - repo: smallbets/userbase-homepage
    branch: master
    hosts:
      - userbase.com
      - www.userbase.com
    app_port: 80
    healthcheck_path: /up
    healthcheck: https://userbase.com/up
Field Default Meaning
repo required GitHub repository as owner/repo.
name repository name Lowercase app name used for checkout, service, image, and logs.
branch repo default branch Only pushes to this branch deploy.
path /srv/repos/<app> Checkout directory on the server.
hosts empty Public hostnames for generated Kamal proxy config.
app_port 80 Port your container listens on.
healthcheck_path /up Container healthcheck path used by generated Kamal config.
healthcheck empty External healthcheck URL used by Single Server after deploys and in status checks.

Server Environment

Variable Default Purpose
SINGLESERVER_CONFIG /etc/singleserver/apps.yml Main app config path.
SINGLESERVER_STATE_DIR /etc/singleserver Directory for GitHub App credentials and Single Server state.
SINGLESERVER_PORT 8787 Local daemon port. The daemon listens on 127.0.0.1.
SINGLESERVER_PUBLIC_URL init-managed Public webhook/setup URL. The target UX should hide this behind singleserver init.
SINGLESERVER_SETUP_TOKEN init-managed One-time setup token for provider connection pages.

App Contract

Your apps need a Dockerfile, one web port, and a health endpoint. They do not need GitHub Actions workflows, deploy keys, repo-level deploy secrets, repo-level webhooks, or a Kamal config.

Dockerfile → Your repo defines how it builds
port 80 → Default app port, override with --app-port
/up → Default healthcheck path, override with --healthcheck-path

Dockerfile Examples

Static site:

FROM nginx:alpine
COPY dist /usr/share/nginx/html

Node or Bun app:

FROM oven/bun:1
WORKDIR /app
COPY package.json bun.lock ./
RUN bun install --frozen-lockfile
COPY . .
ENV PORT=3000
EXPOSE 3000
CMD ["bun", "start"]
singleserver add https://github.com/you/my-node-app --app-port 3000

Deploys

Whenever you push to GitHub, your server fetches the commit, builds it locally, and uses Kamal to roll it out with zero downtime.

git push → GitHub sends a webhook to your server
allowlist → apps.yml decides whether this repo can deploy
checkout → Your server fetches the exact pushed SHA
build → Kamal builds locally with generated or repo-provided deploy config
live → Kamal rolls it out and Single Server checks health
status → GitHub gets pending, success, or failure commit status

Files On Your Server

Path Purpose
/etc/singleserver/apps.yml App allowlist and config.
/etc/singleserver/singleserver.env Daemon environment.
/etc/singleserver/github-app.json GitHub App metadata.
/etc/singleserver/github-app.private-key.pem GitHub App private key.
/srv/repos/<app> Git checkout for each app.
/srv/storage/<app> Persistent storage for app data.
journalctl -u singleserver.service Daemon and deploy logs.

Build Order

Area Contract
Blank server setup install.sh and singleserver init should make a new Linux server ready in one guided flow.
GitHub and Cloudflare connection init should wrap provider setup. github connect and cloudflare connect should exist as repair commands.
App onboarding singleserver add https://github.com/owner/repo should infer sensible defaults, configure routing, deploy immediately, and show the live URL.
Domains singleserver domains should add, remove, list, and verify app domains through Cloudflare Tunnel.
Observability status, doctor, and logs should make deploy state, runtime logs, DNS, proxy, disk, Docker, and healthchecks obvious.
Secrets and storage env, storage, backup, and restore should make SQLite-backed apps practical on one server.
Lifecycle remove and upgrade should be first-class commands with clear confirmation and rollback paths.

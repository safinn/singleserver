The Single Server command contract.

Single Server itself is a small Go binary: no Node runtime, no database, and no per-repo runner. It runs on your server, receives GitHub webhooks, checks pushes against apps.yml, fetches the exact commit, and uses Kamal to build and roll out each app.

Quickstart

Start with any Linux server from Hetzner, DigitalOcean, AWS, Azure, or another provider. Run every command over SSH on that server.

SSH to your Linux server.

ssh root@203.0.113.10

Install the Single Server CLI.

curl -fsSL https://singleserver.com/install.sh | sh

Initialize your server.

singleserver init

Add your apps.

singleserver add https://github.com/you/my-app

Check what is running.

singleserver status

After that, pushing to the configured branch should deploy automatically.

Concepts

  • Your server The server is the deploy control plane. The Single Server daemon, Docker, Kamal, cloudflared, Git checkouts, app config, deploy credentials, persistent storage, and logs all live on this one machine.
  • GitHub App GitHub is the event source, not the build runner. The GitHub App sends push webhooks to your server and gives Single Server short-lived installation tokens to fetch repositories and write commit statuses.
  • apps.yml This file is the deployment allowlist and the central app registry. Even if the GitHub App can see many repositories, Single Server only deploys repositories listed here.
  • App containers Every app runs in its own Docker container. Single Server prepares the checkout and generated deploy config; Kamal builds the image, starts the new container, healthchecks it, and swaps traffic.
  • Cloudflare Tunnel Public HTTP traffic is expected to enter through Cloudflare Tunnel. App domains map to tunnel routes, so the server does not need to expose a public reverse proxy or manage public TLS certificates directly.
  • Central secrets Deploy credentials and app secrets are server-side concerns. App repositories should not need deploy keys, GitHub Actions secrets, or repository-specific deployment workflows.

Commands

Setup

Install curl -fsSL https://singleserver.com/install.sh | sh

Installs Docker, Kamal, cloudflared, the singleserver CLI, the singleserver.service daemon, and base config files. The installer should be safe to rerun.

Initialize your server singleserver init

Connects Cloudflare Tunnel and GitHub, creates required secrets under /etc/singleserver, starts the daemon, validates the local setup, and finishes with singleserver doctor.

Repair GitHub connection singleserver github connect

Opens or prints the GitHub App setup URL, requests repository contents read and commit statuses write, saves the app credentials on the server, and keeps apps.yml as the deployment allowlist.

Repair Cloudflare connection singleserver cloudflare connect

Creates or reuses a Cloudflare Tunnel, saves credentials on the server, verifies the tunnel is running, and prepares DNS/routing for app domains.

Apps

Add an app singleserver add <github-url> [options]

Adds a GitHub repository, verifies app access, checks for a Dockerfile, writes apps.yml, validates the generated Kamal config, configures app routing, deploys the current branch tip, runs the healthcheck, and prints the live URL.

Options for singleserver add

  • <github-url>Required. A URL like https://github.com/owner/repo. Single Server stores it as owner/repo.
  • --branch <name>Deploy only this branch. Defaults to the repository default branch.
  • --app-port <port>Container port used by the generated Kamal proxy. Defaults to 80.
  • --healthcheck-path <path>Path Kamal uses for container healthchecks. Defaults to /up.
  • --healthcheck <url>External URL Single Server checks after deploys and in status/doctor.
  • --name <app>Override the app name. Defaults to the repository name lowercased.
  • --no-deployConfigure the app but wait for the next push or manual deploy.
  • --dry-runValidate the app and print the changes without writing apps.yml.
  • --path <dir>Advanced checkout path override. Defaults to /srv/repos/<app>.
  • --host <domain>Advanced host override for generated Kamal proxy config. Can be repeated. Most domain work should use singleserver domains.
Deploy manually singleserver deploy <owner/repo> [ref]

Deploys a configured app immediately. If ref is omitted, it deploys the configured branch or the repository default branch. Manual deploys use the same checkout, build, rollout, healthcheck, and GitHub status path as push-triggered deploys.

List apps singleserver list

Prints configured apps, their repositories, branches, domains, healthcheck URLs, and current health state.

Check status singleserver status

Summarizes daemon health, app deploy status, app domains, runtime state, and healthchecks. This is the quick operational view.

Run diagnostics singleserver doctor [app]

Runs deeper checks for daemon health, config validity, GitHub App access, generated deploy config, checkout state, recent deploy outcome, disk, Docker, proxy, DNS, and healthchecks.

View logs singleserver logs [app] [--follow] [--runtime]

Shows deploy logs by default. Filter by app name when provided, use --runtime for app container logs, and use --follow to stream.

Inspect deploy config singleserver render-deploy <owner/repo>

Prints the generated Kamal deploy.yml for a configured app. Use this when you want to see exactly what conventions produced before a deploy.

Remove an app singleserver remove <app>

Removes an app from config, stops containers, removes routes, optionally removes DNS records, and asks before deleting persistent storage.

Management

Manage domains singleserver domains add <app> <domain>

Adds, removes, lists, and verifies app domains. Domain changes update central config, Cloudflare DNS, and tunnel routing.

Manage environment singleserver env set <app> KEY=value

Sets, lists, and removes app environment variables and secrets on the server. Secrets stay out of GitHub repositories.

Enable storage singleserver storage enable <app> --mount /storage

Creates persistent storage for SQLite databases, uploads, and other app-owned files, then mounts it into the app container.

Back up an app singleserver backup <app>

Creates an app backup from its configured persistent storage. SQLite apps should have a first-class, low-drama backup path.

Restore an app singleserver restore <app> <backup-id>

Restores an app backup after confirmation and restarts the app with the restored data.

Upgrade Single Server singleserver upgrade

Downloads the latest release, restarts the service, and runs singleserver doctor to verify the server after the upgrade.

Configuration

Single Server should be convention over configuration. Most apps should be added with only their GitHub URL; /etc/singleserver/apps.yml exists as the central override file when an app needs different defaults.

Conventions

  • Repository A GitHub URL like https://github.com/owner/repo is normalized to owner/repo in apps.yml.
  • App name The app name defaults to the repository name, lowercased. It is used for the checkout path, generated Kamal service name, image name, container names, logs, and storage paths.
  • Branch Deploys default to the repository default branch. Set branch only when production deploys should follow a different branch.
  • Checkout Repositories are checked out under /srv/repos/<app>. Single Server fetches the exact pushed SHA before each deploy.
  • Build The app repository owns its Dockerfile. Single Server does not generate a Dockerfile for the app.
  • Kamal config Single Server generates Kamal config from conventions. A repo-level config/deploy.yml is only for apps that need to opt out of generated config.
  • Port Apps listen on port 80 by default. Use --app-port or app_port when the container listens somewhere else.
  • Healthcheck The default container healthcheck path is /up. External healthchecks should use the app domain plus the healthcheck path.
  • Domains Domains belong to apps, not the host. Domain commands should update Cloudflare DNS, tunnel routing, and app config in one place.
  • Secrets and storage Secrets and persistent app data live on the server. Repositories should not need deploy secrets, deploy keys, or repo-specific workflows.

Apps Config

The shortest app entry is just a repository name.

apps:
  - dvassallo/sillyface-games
  - dvassallo/fullsend

Use an object only when an app needs overrides.

apps:
  - repo: smallbets/userbase-homepage
    branch: master
    hosts:
      - userbase.com
      - www.userbase.com
    app_port: 80
    healthcheck_path: /up
    healthcheck: https://userbase.com/up
Field Default Meaning
repo required GitHub repository as owner/repo.
name repository name Lowercase app name used for checkout, service, image, and logs.
branch repo default branch Only pushes to this branch deploy.
path /srv/repos/<app> Checkout directory on the server.
hosts empty Public hostnames for generated Kamal proxy config.
app_port 80 Port your container listens on.
healthcheck_path /up Container healthcheck path used by generated Kamal config.
healthcheck empty External healthcheck URL used by Single Server after deploys and in status checks.

App Contract

Your apps need a Dockerfile, one web port, and a health endpoint. They do not need GitHub Actions workflows, deploy keys, repo-level deploy secrets, repo-level webhooks, or a Kamal config.

Dockerfile → Your repo defines how it builds
port 80 → Default app port, override with --app-port
/up → Default healthcheck path, override with --healthcheck-path

Dockerfile Examples

Static site:

FROM nginx:alpine
COPY dist /usr/share/nginx/html

Node or Bun app:

FROM oven/bun:1
WORKDIR /app
COPY package.json bun.lock ./
RUN bun install --frozen-lockfile
COPY . .
ENV PORT=3000
EXPOSE 3000
CMD ["bun", "start"]
singleserver add https://github.com/you/my-node-app --app-port 3000

Deploys

Whenever you push to GitHub, your server fetches the commit, builds it locally, and uses Kamal to roll it out with zero downtime.

git push → GitHub sends a webhook to your server
allowlist → apps.yml decides whether this repo can deploy
checkout → Your server fetches the exact pushed SHA
build → Kamal builds locally with generated or repo-provided deploy config
live → Kamal rolls it out and Single Server checks health
status → GitHub gets pending, success, or failure commit status

Files On Your Server

Path Purpose
/etc/singleserver/apps.yml App allowlist and config.
/etc/singleserver/singleserver.env Daemon environment.
/etc/singleserver/github-app.json GitHub App metadata.
/etc/singleserver/github-app.private-key.pem GitHub App private key.
/srv/repos/<app> Git checkout for each app.
/srv/storage/<app> Persistent storage for app data.
journalctl -u singleserver.service Daemon and deploy logs.

Build Order

Area Contract
Blank server setup install.sh and singleserver init should make a new Linux server ready in one guided flow.
GitHub and Cloudflare connection init should wrap provider setup. github connect and cloudflare connect should exist as repair commands.
App onboarding singleserver add https://github.com/owner/repo should infer sensible defaults, configure routing, deploy immediately, and show the live URL.
Domains singleserver domains should add, remove, list, and verify app domains through Cloudflare Tunnel.
Observability status, doctor, and logs should make deploy state, runtime logs, DNS, proxy, disk, Docker, and healthchecks obvious.
Secrets and storage env, storage, backup, and restore should make SQLite-backed apps practical on one server.
Lifecycle remove and upgrade should be first-class commands with clear confirmation and rollback paths.

Single Server turns one Linux box into the place where all your apps live. This page covers everything it does on your behalf and every command you might use. The commands are few and the conventions are predictable, so most of this you'll read once and never need again.

Quickstart

You need a fresh Linux box (Ubuntu, Debian, Amazon Linux 2023, or Rocky Linux 9, on amd64 or arm64) from any provider, root access, and about five minutes. Every command runs on the server.

SSH into the box.

ssh root@<server_ip>

Run the interactive installer.

curl -fsSL https://singleserver.com/install.sh | sh

Add an app.

singleserver add https://github.com/you/your-app

See where things stand.

singleserver status

The installer puts Docker, Kamal, Tailscale, cloudflared, and the singleserver binary on the box, then walks you through three connections: a Tailscale login, a Cloudflare API token, and a one-click GitHub App setup page. Each prompt explains itself, and the installer is safe to rerun if you bail halfway.

From here on, every push to the configured branch deploys automatically. That was the whole setup.

The big picture

There's no magic in here. Single Server is boring, proven tools wired together, plus one small Go binary. Knowing how the pieces relate makes everything else on this page obvious.

Your server is the platform.

The daemon, the CLI, Docker, Kamal, the git checkouts, app config, secrets, persistent storage, backups, and logs all live on one machine. There is no control plane in somebody else's cloud and no dashboard to host. When you want to know what's going on, you SSH in and ask.

One binary, two jobs.

The singleserver binary runs as a systemd daemon that deploys your apps, and doubles as the CLI you use over SSH. Both read and write the same server-side state, so the CLI always reflects what the daemon actually did.

apps.yml is the allowlist.

Installing the GitHub App on a repository only grants access. Nothing deploys unless the repo is listed in /etc/singleserver/apps.yml, and pushes deploy only when they land on the app's configured branch. Everything else GitHub sends gets politely ignored.

GitHub triggers, your server deploys.

During setup, Single Server creates a GitHub App for you. That's GitHub's own mechanism for giving Single Server limited, revocable access to your repos: it sends signed push events, hands the daemon short-lived tokens to fetch the pushed commit, and lets the daemon mark each commit as pending, deployed, or failed. No deploy keys, no GitHub Actions secrets, no per-repo webhooks.

Tailscale is the private way in.

Tailscale SSH gets you into the box without exposing SSH to the public internet. Tailscale Funnel gives GitHub a public URL to reach the daemon's webhook endpoint, so the server itself never needs a domain name or a DNS record.

Cloudflare is the public way in.

App domains point at Cloudflare, not at your server. Single Server creates proxied DNS records and a Cloudflare Tunnel, so visitors get TLS, CDN, and DDoS protection at Cloudflare's edge while traffic reaches your apps through the tunnel. Kamal's proxy binds to loopback only, and your server's IP stays hidden.

Docker and Kamal do the heavy lifting.

Every app runs in its own container, built on the server and pushed through a local registry. Kamal starts the container with the new release and flips traffic over once it's healthy, so deploys land with zero downtime and one app's dependency mess can't leak into another.

What a push does

This is the loop you'll trigger hundreds of times without thinking about it. For a small app, the whole chain lands in under 5 seconds.

git push → GitHub sends a signed push event to your server through Tailscale Funnel.
verify → The daemon checks the signature, then checks the repo and branch against apps.yml.
checkout → It fetches the exact pushed commit into /srv/repos/<app>.
build → Docker builds the container image on the server and ships it through the local registry.
live → Kamal starts the new container and flips traffic once it passes the readiness path.
report → The commit gets a pending, success, or failure status on GitHub, with the deploy time.

A few details worth knowing. Deploys for the same app run one at a time, in order, so rapid pushes queue up instead of racing. The first deploy of an app runs kamal setup; every deploy after that runs kamal redeploy. A deploy that exceeds 2 minutes is killed and reported as failed. And if the app has an external healthcheck URL configured, Single Server polls it after the rollout and fails the deploy if it never comes up.

Manual deploys go through exactly the same path. singleserver deploy exists for redeploys, rollbacks, and staged changes, not as a separate mechanism.

Commands

Everything is a subcommand of singleserver, run on the server over SSH. Commands print aligned check lines as they work, so you can see each step succeed or fail. In the usage lines, <angle brackets> mean required and [square brackets] mean optional. Flags are optional unless marked required, with defaults shown where one exists. Run singleserver help anytime for the short version of this section.

Setup and repair

Install or upgrade everything

curl -fsSL https://singleserver.com/install.sh | sh

Installs Docker, Kamal, Tailscale, cloudflared, sqlite3, the singleserver binary, the systemd daemon, a local Docker registry, and a deploy user for Kamal. Then it runs first-run setup: connecting Tailscale, Cloudflare, and GitHub, and finishing with doctor. Rerunning it is always safe; it skips what's already done.

  • TAILSCALE_AUTHKEYSet for unattended installs so Tailscale joins without a browser login.
  • CLOUDFLARE_API_TOKENSet for unattended installs so Cloudflare connects without a prompt. The token needs DNS and tunnel permissions.
  • SINGLESERVER_INSTALL_SKIP_FIRST_RUN=1Install the software but skip the guided setup, for when you want to run the connect commands yourself.

Connect or repair Tailscale

singleserver tailscale connect [--auth-key <key>] [--hostname <name>]

Brings up Tailscale, enables Tailscale SSH, and starts Tailscale Funnel for the daemon's local port. The resulting https://<host>.<tailnet>.ts.net URL becomes the public address GitHub uses for the setup page and webhooks. If Tailscale isn't logged in yet and no auth key is available, it tells you to run tailscale up --ssh first and try again.

  • --auth-key <key>Join the tailnet unattended. Prefer the TAILSCALE_AUTHKEY env var in scripts so the key stays out of shell history.
  • --hostname <name> Default: server hostnameSet the machine name in your tailnet during an unattended join.

Connect or repair Cloudflare

singleserver cloudflare connect [--account <id>] [--tunnel <name>]

Creates or reuses a Cloudflare Tunnel, stores its credentials, installs the cloudflared systemd service, and prunes tunnel routes for domains that no longer belong to any app. After this, every domain you attach to an app gets a proxied CNAME to the tunnel automatically. Reads the API token from CLOUDFLARE_API_TOKEN or the saved state from a previous run.

  • --account <id> Default: auto-detectedPick the Cloudflare account when the token can reach more than one.
  • --tunnel <name> Default: singleserver-<hostname>The Cloudflare Tunnel name.

Connect or repair GitHub

singleserver github connect

Prints the one-time GitHub App setup URL, served by your own daemon through Tailscale Funnel. Opening it in a browser creates the GitHub App with one click and stores its credentials on the server. The app asks for the minimum it needs: repository contents read, commit statuses write, and push events. Install it on all repositories and let apps.yml decide what actually deploys.

Upgrade Single Server

singleserver upgrade

Reruns the installer, restarts the daemon, and finishes with doctor so you know the upgrade landed cleanly. Running apps are not touched.

Print the version

singleserver version

Prints the installed version, commit, and build date.

Apps

Add an app

singleserver add https://github.com/owner/repo [options]

Adds a repository, verifies the GitHub App can reach it, writes the entry to apps.yml, wires up any domains, deploys the current branch tip, and runs doctor. In an interactive session it asks only the questions it can't answer itself: the runtime if there's no Dockerfile, an optional domain, the readiness path, an optional external healthcheck, and whether to deploy now. It then prints the equivalent flag-based command so you can script the same thing next time.

  • --domain <host>Public hostname for the app, like app.example.com. Repeat the flag for more than one. DNS, the tunnel route, and routing follow automatically when Cloudflare is connected.
  • --branch <name> Default: repo default branchDeploy only pushes to this branch.
  • --name <app> Default: repo name, lowercasedOverride the app name. Names must be unique on the server.
  • --runtime static|node|bun Required without a DockerfileGenerate a temporary Dockerfile during deploys. Repos with their own Dockerfile must omit this.
  • --install <cmd> / --build <cmd>Optional install and build commands for generated Node and Bun Dockerfiles, like npm ci and npm run build. Single Server runs exactly what you give it; it doesn't guess.
  • --start <cmd> / --app-port <port> Required for processesStart command and listening port for a generated Node or Bun web process. Neither applies to static output.
  • --static-dir <dir> Default: .Directory of static files to serve with nginx, relative to the repo. For Node/Bun builds it's the build output, like dist.
  • --healthcheck-path <path> Default: / (static: /up)Path Kamal polls to decide the new container is ready.
  • --healthcheck <url>Optional public URL checked after each deploy and by status and doctor. Without it, the external check is reported as assumed.
  • --no-deployWrite the config but wait for the next push or a manual deploy.
  • --yesSkip the interactive prompts and accept defaults, for scripts.

Edit an app

singleserver edit <app|owner/repo|github-url> [options]

Changes an app's settings after it's been added. Run it with no flags in an interactive session and it walks through the current settings, build mode first, then prints the equivalent command. Run it with flags in scripts. Changes deploy immediately unless you pass --no-deploy.

  • --dockerfileSwitch back to the repository Dockerfile and clear any generated runtime settings.
  • --runtime static|node|bunSwitch to a generated Dockerfile, with the same companion flags as add: --install, --build, --start, --static-dir, --app-port.
  • --branch <name>Change which branch deploys automatically.
  • --healthcheck-path <path> Default: / (static: /up)Change the container readiness path.
  • --healthcheck <url> / --no-healthcheckSet or clear the optional external healthcheck URL. Container readiness still runs during deploys either way.
  • --no-deployWrite the config change without applying it to the running app yet.

Deploy or roll back

singleserver deploy <owner/repo|app> [ref]

Deploys a configured app right now, through the same path a push takes. With no ref it deploys the tip of the configured branch. Pass a branch, tag, or commit SHA to deploy exactly that, which is also how you roll back: deploy the last commit that worked.

Inspect the generated Kamal config

singleserver render-deploy <owner/repo|app>

Prints the exact Kamal deploy.yml the conventions would produce for an app, without touching anything. Useful for trust-but-verify moments and for learning what a setting actually changes.

Remove an app

singleserver remove <app> [--delete-repo] [--delete-storage] [--yes]

Removes the app's config entry, its Cloudflare DNS records and tunnel routes, and its containers. The repo checkout and persistent storage stay on disk by default, so removing an app never silently deletes data.

  • --delete-repo --yesAlso delete the checkout under /srv/repos. The --yes is required because it's destructive.
  • --delete-storage --yesAlso delete the persistent storage directory. Same deal.

Watching things

List apps

singleserver list

One line per app: repo, branch, domains, current state, and healthcheck setting.

Check status

singleserver status

The quick operational view: daemon health, then each app's running container, last deploy outcome, and external healthcheck result. This is the command you run when someone says "is it down?"

Read logs

singleserver logs [app] [--runtime] [--daemon] [--follow]

Shows recent deploy logs, optionally filtered to one app. Two switches change the source: --runtime tails the app's container (your app's own output), and --daemon shows the full Single Server journal. Add --follow to stream live.

Run diagnostics

singleserver doctor [app]

The deep check. It walks the entire chain: daemon, config, Docker and buildx, the deploy user and its SSH path, the local registry, disk space, Tailscale and Funnel, the Cloudflare tunnel, its routes and DNS records, the GitHub App and its webhook URL, and then each app's installation, deploy config, checkout, last deploy outcome, and healthcheck. Every line says ok, pending, or failed with the reason. When something is misconfigured, this is where it shows up.

Domains, env vars, storage, backups

Manage domains

singleserver domains add <app> <domain> [--no-deploy]
singleserver domains remove <app> <domain> [--no-deploy]
singleserver domains list [app]
singleserver domains verify [app]

Attaches and detaches public hostnames. add finds the Cloudflare zone that owns the hostname, writes a proxied CNAME to your tunnel, adds the tunnel route, and redeploys the app so Kamal routes the new host. remove undoes all of that. A domain can belong to one app at a time. verify checks resolver DNS and the Cloudflare records when you want reassurance.

  • --no-deployUpdate config and DNS without redeploying the app yet.

Manage env vars

singleserver env set <app> KEY=value
singleserver env list <app>
singleserver env unset <app> KEY

App secrets and config live in a per-app env file on the server, never in the repo. Kamal injects them into the container on the next deploy, so set and unset print the deploy command to run when you're ready to apply.

Enable persistent storage

singleserver storage enable <app> [--mount /storage] [--path /srv/storage/<app>] [--no-deploy]

Containers are disposable; storage is not. This creates a host directory and mounts it into the container at /storage, surviving every deploy. It's the expected home for SQLite databases, uploads, and anything else your app wants to keep.

  • --mount <path> Default: /storageWhere the directory appears inside the container.
  • --path <path> Default: /srv/storage/<app>Where the directory lives on the host.
  • --no-deployStage the storage config without applying it to the running app yet.

Back up and restore

singleserver backup <app>
singleserver restore <app> <backup-id-or-path> --yes [--no-restart]

backup snapshots the app's storage into a timestamped .tar.gz under /srv/backups/<app>. SQLite databases are copied through SQLite's own backup API, so they're consistent even while the app is writing. restore stops the app's containers, swaps the storage directory for the backup, keeps the old directory next to it as a safety copy, and starts the containers again.

  • --yes Required for restoreRestore refuses to run without it, because it replaces the live storage directory.
  • --no-restartRestore the files without touching containers, for when you've already stopped the app yourself.

Configuration

There is one config file: /etc/singleserver/apps.yml. You'll rarely open it, because add, edit, domains, and storage write it for you. But it's plain YAML on your own disk, and editing it by hand is fully supported. Everything not written down falls back to a convention.

The shortest possible app is one line.

apps:
  - you/your-app

That gives you: app name your-app, the repository default branch, a checkout under /srv/repos/your-app, the repo's own Dockerfile, and a generated Kamal config. Use a map when an app needs to differ.

apps:
  - repo: you/your-app
    branch: production
    hosts:
      - app.example.com
      - www.example.com
    runtime: node
    install: npm ci
    build: npm run build
    static_dir: dist
    healthcheck: https://app.example.com/up
    storage:
      mount: /storage

App settings

  • repoRequired. GitHub repository as owner/repo. The only field a conventional app needs.
  • nameDefault: the repository name, lowercased. The app's identity on the server: it names the checkout, the Kamal service, the image, the containers, the logs, and the storage path. Must be unique, so set it when two of your repos share a name.
  • branchDefault: the repository default branch. Only pushes to this branch deploy automatically. Manual deploys can still target any ref.
  • pathDefault: /srv/repos/<name>. Where the server keeps the checkout.
  • hostsDefault: none. Public hostnames. Each one becomes a Kamal proxy host, a proxied Cloudflare CNAME, and a tunnel route. A hostname can belong to one app at a time.
  • runtimeDefault: absent. static, node, or bun. Set only when the repo has no Dockerfile and Single Server should generate one during deploys.
  • install / buildDefault: absent. Optional commands for generated Node and Bun Dockerfiles. Exactly what you write is what runs.
  • startDefault: absent. Start command for a generated Node or Bun web process. Required when runtime is node or bun and there's no static_dir.
  • static_dirDefault: . for runtime: static. Directory served by nginx, relative to the repo. For Node/Bun builds, the build output directory.
  • app_portDefault: 80. The container port that receives traffic. Required for generated Node/Bun processes; always 80 for static output.
  • healthcheck_pathDefault: /, or /up for generated static output. The path Kamal polls before flipping traffic to a new container.
  • healthcheckDefault: absent. Optional external URL checked after deploys and by status and doctor. When absent, the external check is reported as assumed rather than failed.
  • storageDefault: absent until enabled. Persistent storage mapping, with optional path (host, default /srv/storage/<name>) and mount (container, default /storage).

Generated Dockerfiles

A repo with a Dockerfile is always used as-is, no questions asked. For everything else, Single Server generates a Dockerfile from your explicit settings at deploy time, uses it for the build, and deletes it afterward. Nothing is committed to your repo. Three shapes are supported.

# Static files, already built. nginx serves them.
singleserver add https://github.com/you/site --runtime static --static-dir dist

# A build step that outputs static files. Build with Node, serve with nginx.
singleserver add https://github.com/you/site --runtime node --install "npm ci" \
  --build "npm run build" --static-dir dist

# A long-running web process.
singleserver add https://github.com/you/app --runtime bun --install "bun install" \
  --start "bun run start" --app-port 3000

Static output gets an nginx image with sensible behavior: an /up endpoint that answers the readiness check, single-page-app fallback to index.html, and dotfiles blocked. Generated processes run on node:20-alpine or oven/bun:1-alpine, with PORT set to your app port and, for Node, NODE_ENV=production.

Generated Kamal defaults

Repos don't need a Kamal config. If a repo tracks config/deploy.yml, that file wins. Otherwise Single Server writes one for the deploy and removes it after Kamal exits. render-deploy shows you exactly what it generates, and these are the defaults inside it.

  • Service and imageThe app name. One name everywhere makes containers, images, and logs easy to find.
  • Server127.0.0.1, via the deploy user. Kamal keeps its normal SSH deploy model; it just happens to SSH into the same machine.
  • Registry127.0.0.1:5555. A local registry container, so images never leave the box.
  • BuilderLocal Docker builder, server architecture. No cross-compilation, no remote builders.
  • Proxyssl: false, forwarded headers on, bound to loopback. Cloudflare terminates TLS at the edge; the tunnel delivers plain HTTP to Kamal's proxy, which is not reachable from the public internet.
  • ReadinessPoll the healthcheck path every second, 1s timeout. Tight intervals keep zero-downtime swaps fast for small apps.
  • TimeoutsDeploy 10s, drain 1s. Tuned for small containers that boot fast. A repo-level config/deploy.yml is the escape hatch for apps that need different numbers.

The app contract

If your app serves HTTP on a port, Single Server can run it. That's the whole contract. Everything below is optional, there for the day you need it.

HTTP → Serve requests on one port. Port 80 by convention, or tell Single Server which.
env vars → Need config or secrets? Read env vars. Set them with singleserver env.
/storage → Need files that outlive deploys? Write under the mount from singleserver storage.
readiness → Want smarter swaps? Serve a readiness path and point --healthcheck-path at it.

Notably absent: GitHub Actions workflows, deploy keys, repo secrets, webhook setup, Kamal config, and TLS certificates. Apps stay application code.

Files on your server

No hidden state. Everything Single Server knows is in plain files you can read, back up, or move to a new box.

  • /etc/singleserver/apps.ymlThe app allowlist and all app config.
  • /etc/singleserver/singleserver.envDaemon environment: local port, public Funnel URL, setup token.
  • /etc/singleserver/env/<app>.envPer-app env vars and secrets, injected on deploy.
  • /etc/singleserver/github-app.jsonGitHub App ID, slug, and webhook secret.
  • /etc/singleserver/github-app.private-key.pemGitHub App private key, used to mint short-lived repo tokens.
  • /etc/singleserver/tailscale.jsonTailscale hostname and Funnel URL.
  • /etc/singleserver/cloudflare.jsonCloudflare account, tunnel, and token state.
  • /etc/cloudflared/singleserver.ymlTunnel ingress routes, one per app domain.
  • /srv/repos/<app>Git checkout per app, reset to the exact pushed commit on every deploy.
  • /srv/storage/<app>Persistent app storage, mounted into the container.
  • /srv/backups/<app>Timestamped storage backups.
  • journalctl -u singleserver.serviceDaemon and deploy logs, also reachable through singleserver logs.

When something breaks

Start with doctor.

Run singleserver doctor. It checks the whole chain end to end and tells you which link failed and why, in one screen. Most problems stop here.

A deploy failed.

The commit on GitHub shows the failure status with the reason. On the server, singleserver logs <app> has the full build and rollout output, and singleserver logs <app> --runtime shows what your app printed if the container started and then misbehaved. Once you've fixed the cause, push again or rerun singleserver deploy <app>.

A connection broke.

The connect commands are also the repair commands. singleserver tailscale connect, singleserver cloudflare connect, and singleserver github connect each re-establish their piece without disturbing the others, and they're safe to run on a hunch.

You're stuck anyway.

It's standard parts underneath: docker ps, journalctl -u singleserver.service, and the cloudflared and tailscaled services behave exactly as their own docs describe. Or open an issue on GitHub.
